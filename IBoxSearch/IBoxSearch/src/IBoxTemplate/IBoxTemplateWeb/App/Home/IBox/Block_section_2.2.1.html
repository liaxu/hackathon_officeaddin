Standard text and formatting

Follow all authoring guidelines described in General Authoring Guidelines. For additional information on standard text and formatting for this section, see the standards listed at the end of the Standard text and formatting for section 2 Messages.

If the protocol does not use XML namespaces, delete this section.

Use a consistent namespace prefix throughout this specification. Consistently use the namespace prefix assigned in this section for all elements and attributes that have an XML namespace, with the following exception:

The specification MAY choose one namespace that has already been assigned a prefix as the primary namespace for the specification. When referring to definitions from the primary namespace, the prefix MAY be omitted. If a primary namespace is used, then the prefix value of "(none)" MUST be used in the table below. The namespace chosen as the primary namespace SHOULD be the same as the targetNamespace value of the WSDL and/or XML Schema documents. This will allow definitions specific to this protocol to be referenced without a prefix and therefore making the specification easier to read. These definitions can be message names, Port Type names, XML schema element definitions, XML schema attribute definitions, or XML schema type.

If the XML namespace is defined by another specification, include a normative reference to the specification that defines the namespace.

If the XML namespace is defined by this specification, leave the reference blank.

Use the following text and table format when presenting the list of XML namespaces:

This specification defines and references various XML namespaces using the mechanisms specified in [XMLNS]. Although this specification associates a specific XML namespace prefix for each XML namespace that is used, the choice of any particular XML namespace prefix is implementation-specific and not significant for interoperability.

Required elements

  • Use a chosen namespace prefix consistently throughout this specification.

  • List prefixes in the table format displayed above.

  • Include a reference in the prefix table to other documents that define the XML namespace.

  • Leave the reference blank in the prefix table if the XML namespace is defined by this specification.

Inappropriate elements

  • Lowercase versions of normative terms; for example, must, should, and may

Checklist

  • Uses a chosen namespace prefix consistently throughout this specification

  • Lists prefixes in the table format displayed above

  • Includes a reference in the prefix table to other documents that define the XML namespace

  • Leaves the reference blank in the prefix table if the XML namespace is defined by this specification

  • Correctly uses and capitalizes normative terms, as specified in RFC 2119

  • Does not use the lowercase versions of normative terms

  • Does not use the technical document short name ([MS-xxxx]) to refer to the protocol

  • Uses informative citations sparingly

  • Cites only approved references by normative and informative citations

  • Correctly formats pseudocode or code examples

  • Presents product behavior notes in the correct format <#> and links to the correct note in the Product Behavior appendix.

  • Uses the correct product name in references to specific products.

  • Provides a figure number and caption for each diagram, figure, or graphic

  • Correctly formats any tables that specify message syntax

  • Footnotes are only used if you are not authoring in GenDox; any footnote text appears immediately after the table and clearly indicates which information it applies to.
