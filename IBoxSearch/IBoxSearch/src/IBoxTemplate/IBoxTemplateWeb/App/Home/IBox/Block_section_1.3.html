Standard text and formatting

Follow all authoring guidelines described in General Authoring Guidelines.

If a diagram is included in this section, it should use a subset of the UML 2 sequence diagram format, which is available in Microsoft Visio 2003 and subsequent versions.

In the diagram, use a rectangle with an underlined name label to represent a communicating entity. Use a dotted line to represent the timeline of the entity's events over time.

To represent a message, use a diagonal line that connects to the relevant entities' timelines, with a label that contains the name of the message or a description of the message.

Required elements

  • Briefly describe the purpose of the format, algorithm, or protocol and the problem that the format, algorithm, or protocol solves.

  • Provide a high-level scenario for using the format, algorithm, or protocol.

  • Describe important, but not necessarily obvious features, of the format, algorithm, or protocol.

Inappropriate elements

  • Implementation details, such as message syntax and message, method, or API names.

  • Normative terms, as specified in RFC 2119; for example, MUST, SHOULD, and MAY.

  • Lowercase versions of normative terms; for example, must, should, and may. Instead, use alternate words such as "are" or "can."

  • Normative citations.

  • Product behavior notes.

  • Footnotes.

  • Code examples.

  • Pseudocode.

  • Product names, unless those names are part of a protocol name.

Checklist

  • Describes the purpose of the format, algorithm, or protocol and the problem that the format, algorithm, or protocol solves.

  • Provides one or more high-level scenarios for using the format, algorithm, or protocol.

  • Enables implementers to understand how the format, algorithm, or protocol operates without reading any other references.

  • Helps implementers design a format, algorithm, or protocol that is roughly equivalent to the one being specified.

  • Describes important, but not necessarily obvious, features of the format, algorithm, or protocol.

  • Does not provide implementation details, such as message syntax, and method, message, and API names.

  • If the format, algorithm, or protocol inherently has different classes of functionality, describes those roles.

  • Provides high-level descriptions of any relevant conceptual data models.

  • Includes sample network diagrams, as necessary, and those diagrams use a subset of the UML 2 sequence diagram format.

  • Provides a figure number and caption for each diagram; for example, "Figure 1: Information path for list schemas"

  • Uses informative citations sparingly.

  • Does not use normative terms, as defined in RFC 2119.

  • Does not use lowercase versions of normative terms.

  • Does not refer to the format, algorithm, or protocol by using the document short name; instead, uses only the full name of the format, algorithm, or protocol, or "this file format" or "this algorithm" or "this protocol."

  • Does not contain product names, unless those names are part of a format, algorithm, or protocol name.

  • Does not contain normative citations.

  • Does not contain pseudocode or code examples.

  • Does not contain footnotes or product behavior notes.
