Standard text and formatting

Note:  Documents that are created using the GenDox authoring tool will have this section formatted automatically as long as each informative reference is styled using the GenDox "Informative Reference" style at least once in the body of the document.

Follow all authoring guidelines described in General Authoring Guidelines.

The syntax for references is very specific and must be used for all citations and in all specifications. The Interoperability team is responsible for defining and providing the text of a citation when they review and approve a reference request.

If a Windows Communication Foundation (WCF) appendix is present, the following reference to service contracts MUST be included:

[MSDN-WCFSC] Microsoft Corporation, "Designing Service Contracts", http://msdn.microsoft.com/en-us/library/ms733070.aspx

Apply the following formatting rules to citations in this section:

  • If the specification does not cite any informative references or use any glossary terms, do not delete this section. Instead, ensure that the first and only sentence in this section is "None."

  • List references in ascending alphanumeric order by the technical document short name.

  • Use only the exact text specified and approved by the Interoperability team, as indicated in the GenDox Shared Content pane References tab.

  • Enclose the short name of the reference in square brackets ([]).

  • Append a period (.) to all references that DO NOT end in a URL. Reference entries that end in a URL should not contain end punctuation.

  • Use FWLinks for links to documents that are not Open Specifications technical documents. Although the URL that displays in the references section is the URL of the link target, the actual link is an FWLink for links to documents that are not Open Specifications documents. The Interop references editor provides the FWLink when approving a reference request. For example, the destination URL for the pseudocode standard is http://go.microsoft.com/fwlink/?LinkId=89842. If the destination is an Open Specifications technical document, the destination URL should be a .pdf file.

  • If there is an official standards designation for the technical document, omit spaces in the bracketed citation; for example, RFC2119, 802.1D-2004, and FIPS199.

  • If there isn't an official standards designation for the technical document, such as a recommendation by the W3C, use an abbreviated form of the technical document title for the bracketed citation; for example, W3C-PNG, HTML, and CSS-LEVEL2.

  • Citing KB articles outside of the Product Behavior appendix is discouraged. Verify that citations outside of the Product Behavior appendix are not implementation-specific.

Required elements

  • If the specification does not cite any informative references or use any glossary terms, ensure that the first and only paragraph in this section is "None."

  • Use FWLinks as the destination URLs for links to documents that are not Office technical documents. The Office Interoperability team provides FWLinks for you when they approve a reference request.

Optional elements

  • None.

Checklist

  • If the specification does not cite any informative references, the first and only paragraph in this section is "None."

  • Cites [MS-GLOS], [MS-OFCGLOS], [MS-OXGLOS], or the master reference document for your document set if the specification uses terms that are defined in any of those glossaries.

  • All references have been reviewed and approved by the Interoperability team.

  • Includes all references that are cited in the specification and unused references have been deleted from this section.

  • Lists all references that are cited informatively in the specification.

  • Lists references in ascending alphanumeric order by short name.
