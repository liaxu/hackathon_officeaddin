Standard text and formatting

Follow all authoring guidelines described in General Authoring Guidelines.

For SOAP specifications, if it is discovered that the schema is less restrictive than the protocol, include the following text:

In the following sections, the schema definition might be less restrictive than the processing rules imposed by the protocol. The WSDL in this specification matches the WSDL that shipped with the product and provides a base description of the schema. The text that introduces the WSDL specifies additional restrictions that reflect actual Microsoft product behavior. For example, the schema definition might allow for an element to be empty, null, or not present, but the behavior of the protocol as specified restricts the same elements to being non-empty, not null, and present.

Exchange: Use the following boilerplate text for Web service specifications:

In the following sections, the schema definition might differ from the processing rules imposed by the protocol. The WSDL in this specification provides a base description of the protocol. The schema in this specification provides a base description of the message syntax. The text that specifies the WSDL and schema might specify restrictions that reflect actual protocol behavior. For example, the schema definition might allow for an element to be empty, null, or not present, but the behavior of the protocol as specified restricts the same elements to being non-empty, not null, or present.

Otherwise, do not add any text after the Messages heading. The heading for this section should be followed immediately by the heading for section 2.1.

The following standards apply to all subsections within the Messages section:

  • All programming elements (messages, fields, elements, attributes, parameters, data types, etc.) must be formatted according to the Programming Element Formatting Guidelines.

  • For inline comments in code examples of any type of protocol, use phrases only. Do not append any closing punctuation. Do not capitalize the first word of the phrase.

  • For block-style code comments, use sentences or phrases. If the comment is a complete sentence, append closing punctuation, such as a period, to the end of it. If it is a phrase, do not append any closing punctuation.

Required elements

  • Include introductory text if the schema is less restrictive than the protocol.

Optional elements

  • None.

Inappropriate elements

Text immediately after the section heading; however, introductory text following the section heading is allowed if the schema is less restrictive than the protocol, as indicated in the "Required elements" section.

Checklist
