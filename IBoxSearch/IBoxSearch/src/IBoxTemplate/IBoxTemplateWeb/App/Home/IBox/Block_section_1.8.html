Standard text and formatting

Follow all authoring guidelines described in General Authoring Guidelines.

If there are no vendor-extensible fields for the protocol, do not delete this section. Instead, ensure that the first and only sentence in this section is "None." Typically, there are no vendor-extensible fields for stored procedures (sprocs).

If the protocol returns HRESULTs, include the following text:

This protocol uses HRESULT values as defined in [MS-ERREF] section 2.1. Vendors can define their own HRESULT values, provided that they set the C bit (0x20000000) for each vendor-defined value, indicating the value is a customer code.

If the protocol returns Win32 error codes, include the following text:

This protocol uses Win32 error codes as defined in [MS-ERREF] section 2.2. Vendors SHOULD reuse those values with their indicated meaning. Choosing any other value runs the risk of a collision in the future.

If the protocol returns NTSTATUS values, include the following text:

This protocol uses NTSTATUS values as defined in [MS-ERREF] section 2.3. Vendors can choose their own values for this field, as long as the C bit (0x20000000) is set, indicating it is a customer code.

Exchange Server technical documents do not reference section numbers in Microsoft Open Specifications technical documents that are not part of the Exchange Server Open Specifications documentation set.

Required elements

  • If there are no vendor-extensible fields for the protocol, the first and only paragraph in this section is "None."

  • If a field is a string field, state whether it is a UNICODE or ASCII field.

  • If a field is reserved or unused, state the required setting of the reserved, unused, or padding field; for example, "MUST be set to zero when sending and ignored when receiving."

  • If the protocol returns HRESULTs, Win32 error codes, or NTSTATUS values, use the correct standard text to indicate as much.

Inappropriate elements

  • Error codes that affect the behavior of or change the state of a protocol; instead, document these error codes later in the specification

  • Error codes that can be returned by a Microsoft API that is both available to third parties and documented publicly by Microsoft, on MSDN or an equivalent public location; instead, document these error codes in the product behavior appendix

  • Error values that are handled specially by Microsoft applications or components of a Microsoft product or technology; instead, document the handling in an error code list in an appendix

  • Lowercase versions of normative terms; for example, must, should, and may

  • Footnotes

  • Code examples

  • Product names, unless those names are part of a protocol name

Checklist

  • If there are no vendor-extensible fields for the protocol, the first and only paragraph in this section is "None."

  • Specifies all vendor-extensible fields.

  • Explains the process for acquiring values for all vendor-extensible fields.

  • Uses the correct standard text for cases in which the protocol returns HRESULTs, Win32 error codes, or NTSTATUS values.

  • Discusses any off-box uniqueness constraints.

  • Normative terms, as specified in RFC 2119, are used and capitalized correctly.

  • Does not use the lowercase versions of normative terms.

  • Does not use the technical document short name ([MS-xxxx]) to refer to the protocol.

  • Normative citations cite only approved normative references.

  • Presents product behavior notes in the correct format <#> and links to the correct note in the Product Behavior appendix.

  • Uses the correct product name in references to specific products.

  • Includes accurate and illustrative diagrams, figures, or other art as necessary

  • Provides a figure number and caption for each diagram, figure, or graphic; for example, "Figure 1: This protocol in relation to other protocols"

  • Does not contain pseudocode or code examples.

  • Does not contain tables.

  • Does not contain footnotes.
