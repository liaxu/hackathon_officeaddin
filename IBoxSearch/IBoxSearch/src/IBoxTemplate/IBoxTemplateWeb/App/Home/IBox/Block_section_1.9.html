Standard text and formatting

Follow all authoring guidelines described in General Authoring Guidelines.

If there are no standards assignments for the protocol or algorithm, delete the table of assignments and ensure that the first and only paragraph in this section is "None."

If standards assignments apply to the protocol or algorithm, specify them in a table that contains the following columns:

  • Parameter - Identifies the assignment

  • Value - Specifies the value that is used for the assignment

  • Reference - Provides a link to the corresponding documentation or assignment registry for the assignment

The table should adhere to the following format.

Required elements

  • If there are no standards assignments for the protocol or algorithm, delete the table of assignments and ensure that the first and only paragraph in this section is "None."

  • Answer the following questions:

    • Have any standard assignments been received for this protocol or algorithm?

    • Specifically, does this protocol have a standard TCP or UDP port, Service Name, or other type of identifier? If so, list them here and provide a reference to IANA documentation or to the documentation of some other assignment authority. The use of such values should be described in the appropriate section below.

  • Identify pipe name values as pipe\name.

  • For RPC-based protocols, include a reference for their UUID to [C706] section A.2.5.

  • If an RPC-based protocol uses a well-known endpoint, reference the endpoint here and describe it in Transport.

  • If an RPC-based protocol uses dynamic endpoints, include an informative citation to [C706], as described in Transport.

  • For REST protocols, specify TCP ports in Transport.

  • For REST protocols, specify HTTP status codes in Message Processing Events and Sequencing Rules (REST).

  • If applicable to your protocol, list XML namespaces in Namespaces.

Inappropriate elements

  • Normative terms, as specified in RFC 2119; for example, MUST, SHOULD, and MAY

  • Lowercase versions of normative terms; for example, must, should, and may

  • Product behavior notes

  • Diagrams, figures, or other art

  • Code examples

  • Pseudocode

  • Product names, unless those names are part of a protocol name

Checklist

  • If there are no standards assignments, the first and only paragraph is "None." and the Parameter/Value/Reference table has been removed.

  • Lists and describes any standard TCP or UDP port numbers, service names, RPC UUIDs, named pipe names, or other types of identifiers.

  • References to assignment authorities are valid.

  • Does not use normative terms, as specified in RFC 2119

  • Does not use the lowercase versions of normative terms

  • Does not use the technical document short name ([MS-xxxx]) to refer to the protocol or algorithm

  • Does not contain product behavior notes.

  • Does not contain pseudocode or code examples

  • Does not contain product names, unless those names are part of a protocol or algorithm name

  • Footnotes are only used if you are not authoring in GenDox; any footnote text appears immediately after the table and clearly indicates which information it applies to.
