Standard text and formatting

Note: Technical documents that are created using the GenDox authoring tool will have this section formatted automatically as long as each normative reference is styled using the GenDox "Normative Reference" style at least once in the body of the technical document.

Follow all authoring guidelines described in General Authoring Guidelines.

For references to RFC standards, it is preferable to link to the RFCs hosted on the www.rfc-editor.org site. It is not wrong to link to other RFC hosts such as www.ietf.org but linking to the www.rfc-editor.org site is preferable.

The syntax for references is very specific and must be used for all citations and in all specifications. The Office Interoperability team is responsible for defining and providing the text of a citation when they review and approve a reference request.

Apply the following formatting rules to citations in this section:

  • If the specification does not use any normative terms or contain any normative citations, do not delete this section. Instead, ensure that the first and only sentence in this section is "None."

  • If the specification cites one or more normative references, the first paragraph in this section should be:

    We conduct frequent surveys of the normative references to assure their continued availability. If you have any issue with finding a normative reference, please contact dochelp@microsoft.com. We will assist you in finding the relevant information.

  • If the specification uses normative terms, as defined in RFC 2119, ensure that the list of citations includes:

    [RFC2119] Bradner, S., "Key Words for Use in RFCs to Indicate Requirement Levels", BCP 14, RFC 2119, March 1997, http://www.rfc-editor.org/rfc/rfc2119.txt

    The destination URL for the link to RFC 2119 should be http://go.microsoft.com/fwlink/?LinkId=90317.

  • For REST specifications that include an XML schema, the following normative references MUST be included; if the specification does not include an XML schema, the references are optional.

    [RFC2119] Bradner, S., "Key Words for Use in RFCs to Indicate Requirement Levels", BCP 14, RFC 2119, March 1997, http://www.rfc-editor.org/rfc/rfc2119.txt

    [XMLNS] Bray, T., Hollander, D., Layman, A., et al., Eds. "Namespaces in XML 1.0 (Third Edition)", December 2009, http://www.w3.org/TR/2009/REC-xml-names-20091208/

    [XMLSCHEMA1] Thompson, H.S., Ed., Beech, D., Ed., Maloney, M., Ed., and Mendelsohn, N., Ed., "XML Schema Part 1: Structures", W3C Recommendation, May 2001, http://www.w3.org/TR/2001/REC-xmlschema-1-20010502/

    [XMLSCHEMA2] Biron, P.V., Ed., and Malhotra, A., Ed., "XML Schema Part 2: Datatypes", W3C Recommendation, May 2001, http://www.w3.org/TR/2001/REC-xmlschema-2-20010502/

  • The following are additional references commonly included by RESTful and REST-like protocols:

    [RFC2396] Berners-Lee, T., Fielding, R., and Masinter, L., "Uniform Resource Identifiers (URI): Generic Syntax", RFC 2396, August 1998, http://www.rfc-editor.org/rfc/rfc2396.txt

    [RFC2616] Fielding, R., et al., "Hypertext Transfer Protocol -- HTTP/1.1", RFC 2616, June 1999, http://www.rfc-editor.org/rfc/rfc2616.txt

    [RFC2818] Rescorla, E., "HTTP Over TLS", RFC 2818, May 2000, http://www.rfc-editor.org/rfc/rfc2818.txt

    [XML10] World Wide Web Consortium, "Extensible Markup Language (XML) 1.0 (Third Edition)", February 2004, http://www.w3.org/TR/REC-xml

    [XMLINFOSET] World Wide Web Consortium, "XML Information Set (Second Edition)", February 2004, http://www.w3.org/TR/2004/REC-xml-infoset-20040204

  • If the protocol uses CSDL schema elements, include the following normative reference:

    [MC-CSDL] Microsoft Corporation, "Conceptual Schema Definition File Format".

  • If the protocol uses an open data model, include the following normative reference:

    [MS-ODATA] Microsoft Corporation, "Open Data Protocol (OData) Specification".

  • If the protocol uses ABNF syntax, include the following normative reference:

    [RFC5234] Crocker, D., Ed., and Overell, P., "Augmented BNF for Syntax Specifications: ABNF", STD 68, RFC 5234, January 2008, http://www.rfc-editor.org/rfc/rfc5234.txt

  • For RPC-based protocols, the following normative references MUST be included:

    [C706] The Open Group, "DCE 1.1: Remote Procedure Call", C706, August 1997, http://www.opengroup.org/public/pubs/catalog/c706.htm

    [MS-RPCE] "Remote Procedure Call Protocol Extensions".

  • For DCOM-based protocols, the following normative reference MUST be used:

    [MS-DCOM] Microsoft Corporation, "Distributed Component Object Model (DCOM) Remote Protocol Specification".

  • If the specification discusses class attribute pairs for directory service schema elements, cite the appropriate normative references from the following list:

    • [MS-ADA1] Microsoft Corporation, "Active Directory Schema Attributes A-L"

    • [MS-ADA2] Microsoft Corporation, "Active Directory Attributes M"

    • [MS-ADA3] Microsoft Corporation, "Active Directory Attributes N-Z"

    • [MS-ADSC] Microsoft Corporation, "Active Directory Schema Classes"

    • [MS-ADLS] Microsoft Corporation, "Active Directory Lightweight Directory Services Schema"

  • If the protocol uses an XSD, the following normative references MUST be included:

    [XML10] World Wide Web Consortium, "Extensible Markup Language (XML) 1.0 (Third Edition)", February 2004, http://www.w3.org/TR/2004/REC-xml-20040204/

    [XMLINFOSET] Cowan, J., and Tobin, R., Eds., "XML Information Set (Second Edition)", W3C Recommendation, February 2004, http://www.w3.org/TR/2004/REC-xml-infoset-20040204/

    [XMLNS] Bray, T., Hollander, D., Layman, A., et al., Eds., "Namespaces in XML 1.0 (Third Edition)", W3C Recommendation, December 2009 http://www.w3.org/TR/2009/REC-xml-names-20091208/

    [XMLSCHEMA1] Thompson, H., Beech, D., Maloney, M., and Mendelsohn, N., Eds., "XML Schema Part 1: Structures", W3C Recommendation, May 2001 http://www.w3.org/TR/2001/REC-xmlschema-1-20010502/

    [XMLSCHEMA2] Biron, P.V., Ed. and Malhotra, A., Ed., "XML Schema Part 2: Datatypes", W3C Recommendation, May 2001, http://www.w3.org/TR/2001/REC-xmlschema-2-20010502/

  • If the protocol uses JSON, the following normative references MUST be included:

    [ECMA-404] ECMA International, "The JSON Data Interchange Format", ECMA-404, October 2013, http://www.ecma-international.org/publications/files/ECMA-ST/ECMA-404.pdf

    [JSON-Schema] Internet Engineering Task Force (IETF), "JSON Schema and Hyper-Schema", January 2013 http://json-schema.org/

  • If the specification documents a protocol or file format involving Microsoft SQL Server, the following normative references MUST be included:

    [MSDN-TSQL-Ref] Microsoft Corporation, "T-SQL Language Reference", SQL Server 2005 Books Online (November 2008), http://msdn.microsoft.com/en-us/library/ms189826(SQL.90).aspx

    [Iseminger] Microsoft Corporation, "SQL Server 2000 Architecture and XML/Internet Support", Volume 1 of Microsoft SQL Server 2000 Reference Library, Microsoft Press, 2001, ISBN 0-7356-1280-3, http://www.microsoft.com/mspress/books/5001.aspx

    [MS-TDS] Microsoft Corporation, "MS-Tabular Data Stream Protocol Specification".

  • The following normative references MUST be included for all SOAP-based protocols:

    [WSDL] Christensen, E., Curbera, F., Meredith, G., and Weerawarana, S., "Web Services Description Language (WSDL) 1.1", W3C Note, March 2001, http://www.w3.org/TR/2001/NOTE-wsdl-20010315

    [XMLNS] Bray, T., Hollander, D., Layman, A., et al., Eds., "Namespaces in XML 1.0 (Third Edition)", December 2009, http://www.w3.org/TR/2009/REC-xml-names-20091208/

    [XMLSCHEMA1] Thompson, H.S., Ed., Beech, D., Ed., Maloney, M., Ed., and Mendelsohn, N., Ed., "XML Schema Part 1: Structures", W3C Recommendation, May 2001, http://www.w3.org/TR/2001/REC-xmlschema-1-20010502/

    [XMLSCHEMA2] Biron, P.V., Ed. and Malhotra, A., Ed., "XML Schema Part 2: Datatypes", W3C Recommendation, May 2001, http://www.w3.org/TR/2001/REC-xmlschema-2-20010502/

  • The following are additional references commonly included by SOAP-based protocols. If you do not use the reference in your technical document, remove it from the list below.

    [RFC2396] Berners-Lee, T., Fielding, R., and Masinter, L., "Uniform Resource Identifiers (URI): Generic Syntax", RFC 2396, August 1998, http://www.rfc-editor.org/rfc/rfc2396.txt

    [RFC2616] Fielding, R., et al., "Hypertext Transfer Protocol -- HTTP/1.1", RFC 2616, June 1999, http://www.rfc-editor.org/rfc/rfc2616.txt

    [RFC2818] Rescorla, E., "HTTP Over TLS", RFC 2818, May 2000, http://www.rfc-editor.org/rfc/rfc2818.txt

    [SOAP1.1] Box, D., Ehnebuske, D., Kakivaya, G., Layman, A., Mendelsohn, N., Nielsen, H. F., Thatte, S., and Winer, D., "Simple Object Access Protocol (SOAP) 1.1", May 2000, http://www.w3.org/TR/2000/NOTE-SOAP-20000508/

    [SOAP1.2/1] Gudgin, M., Hadley, M., Mendelsohn, N., Moreau, J., and Nielsen, H.F., "SOAP Version 1.2 Part 1: Messaging Framework", W3C Recommendation, June 2003, http://www.w3.org/TR/2003/REC-soap12-part1-20030624

    [SOAP1.2/2] Gudgin, M., Hadley, M., Mendelsohn, N., Moreau, J., and Nielsen, H.F., "SOAP Version 1.2 Part 2: Adjuncts", W3C Recommendation, June 2003, http://www.w3.org/TR/2003/REC-soap12-part2-20030624

    [XML10] World Wide Web Consortium, "Extensible Markup Language (XML) 1.0 (Third Edition)", February 2004, http://www.w3.org/TR/2004/REC-xml-20040204/

    [XMLINFOSET] World Wide Web Consortium, "XML Information Set (Second Edition)", February 2004, http://www.w3.org/TR/2004/REC-xml-infoset-20040204

  • Use FWLinks as the destination URLs for links to references that are not Open Specifications technical documents. FWLinks are provided by the references editor when a reference is approved.

  • Reference entries that end with a URL SHOULD NOT end with a period.

  • Reference entries that do not end with a URL SHOULD end with a period.

  • List references in ascending alphanumeric order by technical document short name.

  • Use only the exact text specified and approved by the references editor, as indicated in GenDox Shared Content pane References tab.

  • Enclose the short name of the reference in square brackets ([]).

  • Append a period (.) to references that do not end in a URL. Reference entries that end with a URL do not receive end punctuation, per Chicago style.

  • Use FWLinks as the destination URL for links to documents that are not Open Specifications technical documents. For example, the destination URL for RFC 2119 is http://go.microsoft.com/fwlink/?LinkId=90317.

  • If the destination is an Open Specifications technical document, the destination URL should be a .pdf file. For example, the destination URL for MS-OFCGLOS is [MS-OFCGLOS].pdf.

  • If there is an official standards designation for the technical document, omit spaces in the bracketed citation; for example, RFC2119, 802.1D-2004, and FIPS199.

  • If there isn't an official standards designation for the document, such as a recommendation by the W3C, use an abbreviated form of the technical document title for the bracketed citation; for example, W3C-PNG, HTML, and CSS-LEVEL2.

Required elements

  • If the specification neither cites normative references nor uses normative terms, ensure that the first and only paragraph in this section is "None."

  • If the specification cites normative references or uses normative terms, ensure that the first paragraph in this section is the standard text about surveys of normative references.

  • Use FWLinks as the destination URLs for links to documents that are not Open Specifications technical documents.

  • If the specification discusses class attribute pairs for directory service schema elements, cite the appropriate normative references from the following list: [MS-ADA1], [MS-ADA2], [MS-ADA3], [MS-ADSC], and/or [MS-ADLS].

  • If the specification documents a protocol, algorithm, or file format involving Microsoft SQL Server, include [Iseminger], [MS-TDS], and [MSDN-TSQL-Ref] as normative references.

  • If the specification documents a protocol, algorithm, or file format that uses XML data structures, include [WSDL], [XMLNS], [XMLSCHEMA1], and [XMLSCHEMA2] as normative references.

  • For references to RFC standards, it is preferable to use www.rfc-editor.org URLs.

Inappropriate elements

  • Citations for normative references that have not been approved by the references editor.

  • Direct links to documents that are not Open Specifications technical documents; instead, use FWLinks for these links. The references editor provides FWLinks for you when they approve a reference request.

  • Extraneous text that has not been provided by the references editor.

  • References to technical documents that have a minimal or ambiguous relationship to the protocol, algorithm, or file format.

  • Informative citations.

  • Product behavior notes.

  • Footnotes.

  • Glossary terms.

  • Diagrams, figures, or other art.

  • Tables.

  • Code examples.

  • Pseudocode.

  • Product names that are not part of the approved citation for a reference.

Checklist

  • If the specification does not use any normative terms or normative references, the first and only paragraph in this section is "None."

  • All references are approved by the references editor.

  • Includes all references that are cited in the specification and all required references have been cited in the specification. Unused references that are not required have been deleted from this section.

  • Lists all references that are cited normatively in the specification.

  • Lists all normative references that are cited informatively in the specification.

  • Lists references in ascending alphanumeric order by short name.

  • Includes commonly used references such as [MS-DTYP] if content from them is used or cited in the specification.

  • If the specification uses normative terms, as defined in RFC 2119, includes a reference to RFC 2119.

  • Does not contain any inappropriate elements.

  • Reference entries that end with a URL should not contain terminal punctuation.

  • References to RFC standards should use www.rfc-editor.org URLs.
