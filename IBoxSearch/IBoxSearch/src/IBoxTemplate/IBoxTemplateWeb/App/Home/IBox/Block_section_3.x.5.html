Standard text and formatting

Follow all authoring guidelines described in General Authoring Guidelines.

If there are no sequencing rules and no actions are taken in response to messages, do not delete this section. Instead, ensure that the first and only sentence in this section is "None."

If there are sequencing rules and actions are taken in response to messages, the structure of this section varies if multiple types are specified in Messages. Create a 3.x.5.x subsection, "Receiving a <Insert Message Name> Message," for each type of message defined in section 2 of the specification.

Required elements

  • If there are no sequencing rules and no actions are taken in response to messages, ensure that the first and only paragraph in this section is "None."

  • Specify any sequencing rules for the protocol.

  • Describe actions that are taken in response to messages that the protocol receives.

  • Document special error handling cases, including responses to specific error conditions and error codes, and, for generalized responses, the range of possible error codes.

  • If the protocol supports multiple message types, indicate how the protocol determines the type of an incoming message and how it handles each type of message.

  • For protocols that support multiple message types, create a distinct subsection 3.x.5.x for each type of message.

Inappropriate elements

  • Sequencing rules and message information that is not relevant to the current section 3.x

  • Lowercase versions of normative terms; for example, must, should, and may

  • Code examples

  • Pseudocode

  • Product names, unless those names are part of a protocol name

Checklist

  • If there are no sequencing rules and no actions are taken in response to messages, the first and only paragraph in this section is "None."

  • Specifies all sequencing rules.

  • Specifies all actions that are taken in response to messages.

  • Identifies special error handling cases are specifies responses to those cases.

  • For generalized error responses, provides the range of possible error codes.

  • If the protocol supports multiple message types, documents type determination and handling.

  • For protocols that support multiple message types, a distinct subsection 3.x.5.x exists for each type of message.

  • Describes procedures/sequences according to guidelines.

  • Specifies all of the message types that described in section 2.

  • Describes only those behaviors that are applicable to the role defined in the same section 3.x.

  • Does not use code examples or pseudocode.

  • Normative terms, as specified in RFC 2119, are used and capitalized correctly.

  • Does not use the lowercase versions of normative terms.

  • Normative and informative citations cite only approved references.

  • Presents product behavior notes in the correct format <#> and links to the correct note in the Product Behavior appendix.

  • Includes accurate and illustrative diagrams, figures, or other art as necessary.

  • Provides a figure number and caption for each diagram, figure, or graphic.

  • Does not use the technical document short name ([MS-xxxx]) to refer to the protocol

  • Does not contain pseudocode or code examples.

  • Does not contain product names, unless those names are part of a protocol name.

  • Footnotes are only used if you are not authoring in GenDox; any footnote text appears immediately after the table and clearly indicates which information it applies to.
