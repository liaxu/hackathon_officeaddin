Standard text and formatting

Follow all authoring guidelines described in General Authoring Guidelines.

  • If no states are required for the protocol or algorithm to work, do not delete this section. Instead, ensure that the first and only sentence in this section is "None."

  • If any states are required for the protocol or algorithm to work, the first paragraph of this section should be:

    This section describes a conceptual model of possible data organization that an implementation maintains to participate in this <protocol/algorithm>. The described organization is provided to facilitate the explanation of how the <protocol/algorithm> behaves. This specification does not mandate that implementations adhere to this model as long as their external behavior is consistent with that described in this specification.

  • If applicable, the section should then list the required states in the following format:

    state name: A/An/The + <brief description that forms a complete sentence>.

  • For example:

    Base URL: The portion of the Request-URI that matches the base rule in the ABNF, as specified in section 2.2.1.3.1.

    Query Parameters: The parameters in the query component of the Request-URI, as specified in section 2.2.1.3.2 and its subsections.

  • If another protocol or algorithm directly sets or gets the value of abstract data model (ADM) elements, include the following statement and include "(Public)" after the name of the ADM element. Otherwise, omit the statement.

    Note that the abstract interface notation '(Public)' indicates that the abstract data model element can be directly accessed from outside this <protocol/algorithm>.

  • If the protocol or algorithm directly sets or gets the value of ADM elements of another protocol or algorithm, that is, (Public) elements of another protocol or algorithm, call out the references explicitly. For example:

    This protocol includes the following ADM elements, which are directly accessed from <protocol name> as specified in [MS-ABCD] section 3.1.1:

    CurrentCounter

    MaxCounter

    The following elements are specific to this protocol:

    Base URL: The portion of the Request-URI that matches the base rule, as specified in section 2.2.1.3.1.

    Query parameters: The parameters in the query component of the Request-URI, as specified in section 2.2.1.3.2.

    Use the same element name that is used in the referenced technical document. If you cannot use the same name, you must provide a mapping between the names.

Required elements

  • Include the standard text above as the first paragraph if any states are required for the protocol or algorithm to work.

  • Provide a conceptual description of any states, except timers, that must be present in some form for the protocol or algorithm to work.

  • List all of the states that are identified as required in behavior specifications for the protocol or algorithm.

  • Ensure that all of the states that are specified in this section are also mentioned elsewhere in the specification when specifying the behavior of the protocol or algorithm.

Inappropriate elements

  • Prescriptive implementation details

  • Information about states that are not relevant to the current section 2.x or 3.x

  • Information about message-specific structures and structure definitions

  • Descriptions of specific classes that are used

  • Lowercase versions of normative terms; for example, must, should, and may

  • Code examples

  • Pseudocode

  • Product names, unless those names are part of the name of a protocol or format

Checklist

  • If no states are required for the protocol or algorithm to work, the first and only sentence in this section is "None."

  • If any states are required for the protocol or algorithm to work, the first paragraph is the standard statement indicating that the information is conceptual.

  • Describes any states, except timers, that must be present in some form for the protocol or algorithm to work.

  • State descriptions are formatted as state name: A/An/The + <brief description that forms a complete sentence>.

  • Does not provide implementation-specific details.

  • All of the states that are included in this section are specified elsewhere in the specification, and all of the states included in behavior specifications elsewhere are mentioned in this section.

  • Describes only those states that are required and are specified in subsections of the same section 2. x or 3. x.

  • Does not define structures or provide information about message-specific structures.

  • Normative terms, as specified in RFC 2119, are used and capitalized correctly.

  • Does not use the lowercase versions of normative terms.

  • Does not use the technical document short name ([MS-xxxx]) to refer to the protocol or algorithm.

  • Normative and informative citations cite only approved references.

  • Presents product behavior notes in the correct format <#> and links to the correct note in the Product Behavior appendix.

  • Includes accurate and illustrative diagrams, figures, or other art as necessary.

  • Provides a figure number and caption for each diagram, figure, or graphic.

  • Does not contain pseudocode or code examples.

  • Footnotes are only used if you are not authoring in GenDox; any footnote text appears immediately after the table and clearly indicates which information it applies to.
