Standard text and formatting

Follow all authoring guidelines described in General Authoring Guidelines.

Section 1.1 should be organized and terms should be listed in ascending alphabetical order (A to Z). Acronyms and abbreviations can appear in the glossary as the spelled out term with the acronym or abbreviation in parentheses, even if only the acronym is used in the body of the document.

If a term has more than one more definition, list the term along with its multiple definitions, marked with number (1), (2) and so on. For example:

contact: (1) An object of the contact class that represents a company or person whom a user can contact. (2) A person, company, or other entity that is stored in a directory and is associated with one or more unique identifiers and attributes (2), such as an Internet message address or login name.

When a single term is used within the same document with multiple definitions, in the body of the document, each instance should indicate the number of the applicable definition by appending it to the term and enclosing that number in parentheses: for example, contact (1) and contact (2).

The last item in the Glossary section should be the standard entry for normative terms:

MAY, SHOULD, MUST, SHOULD NOT, MUST NOT:These terms (in all caps) are used as described in [RFC2119]. All statements of optional behavior use either MAY, SHOULD, or SHOULD NOT.

In that entry, [RFC2119] should be formatted as a normative citation.

If the specification does not contain any glossary terms or normative terms, delete the standard introduction from the specification.

Required elements

  • Organize and list all terms according to ascending alphabetical order.
  • Include definitions for each term in the list, including multiple definitions marked with definition numbers.
  • Ensure that the last entry is the standard entry for normative terms.
  • Apply bold formatting to all terms.

Inappropriate elements

  • Lowercase versions of normative terms; for example, must, should, and may
  • Product behavior notes
  • Footnotes
  • Diagrams, figures, or other art
  • Tables
  • Code examples
  • Pseudocode

Checklist

  • Introduces the glossary term list with "The following terms are specific to this document:".
  • Alphabetizes terms in ascending order within each term list.
  • Applies sentence capitalization rules to terms.
  • Bold formatting is applied to all terms.
  • If the specification uses any SOAP-related terms, those terms are included in the glossary list.
  • If the specification does not contain any glossary terms, the standard introduction is deleted.
  • If the specification contains one or more normative terms, the last item is a citation to RFC 2119. If the specification does not contain any normative terms, this citation does not appear.
  • Does not use the lowercase versions of normative terms.
  • Uses informative citations sparingly.
  • Does not contain normative citations.
  • Does not contain pseudocode or code examples.
  • Does not contain footnotes or product behavior notes.
  • Does not contain diagrams, figures, or other art.
  • Does not contain tables.
