Standard text and formatting

Follow all authoring guidelines described in General Authoring Guidelines. For additional information on standard text and formatting for this section, see the standards listed at the end of the Standard text and formatting for section 2 Messages.

If the protocol accesses or stores data in the Active Directory store, include section 2.3 (Directory Service Schema Elements) after section 2.2 (Common Message Syntax); otherwise, omit section 2.3.

The protocol accesses the following Directory Service schema classes and attributes listed in the following table(s).
For the syntactic specifications of the following <Class> or <Class><Attribute> pairs, refer to:
Active Directory Domain Services (AD DS) ([MS-ADA1],[MS-ADA2], [MS-ADA3], and [MS-ADSC]).
Or:
Active Directory Lightweight Directory Services (AD LDS) ([MS-ADLS]).

Each class or class attribute pair that is cited in this section must be referenced at least once in section 3 of the specification. There are three potential cases as shown in the table above. Include one or more instances of the above table, according to the appropriate cases as described below:

  • Case 1 - If the protocol accesses all of the non-inherited class attributes, there is no requirement to list them individually; "All" is sufficient.

  • Case 2 - If the protocol accesses all of the non-inherited class attributes, and select inherited attributes, list them after "All."

  • Case 3 - If the protocol accesses a proper subset of inherited and non-inherited class attributes, list each attribute individually, in LDAP display name (ldapDisplayName) order.

Required elements

  • If the protocol does not access or store data in an Active Directory store, delete this section, including the heading.

  • Begin with the standard text that introduces the section, the appropriate normative citations, and tables that list the classes and class attributes that are used by the protocol.

  • Ensure that every class or class attribute pair in this section is described in section 3 of the specification.

  • Use normative terms, as specified in RFC 2119; for example, MUST, SHOULD, and MAY.

Inappropriate elements

  • Citations and links to ADA and ADLS normative references that do not apply to the protocol; revise the standard text to omit references that do not apply

  • Lowercase versions of normative terms; for example, must, should, and may

  • Diagrams, figures, or other art

  • Code examples

  • Pseudocode

  • Product names, unless those names are part of a protocol name

Checklist

  • If the protocol does not access or store data in an Active Directory store, this section, including the heading, is omitted from the specification.

  • The first sentence in the section is "The protocol accesses the following Directory Service schema classes and attributes listed in the following table(s)."

  • Includes the standard text for normative references and cites only those references that apply.

  • Lists, in a properly formatted table, all of the classes and class attributes that are used by the protocol.

  • Every class or class/attribute pair listed in this section is described in section 3 of the specification.

  • Normative terms, as specified in RFC 2119, are used and capitalized correctly.

  • Does not use the lowercase versions of normative terms.

  • Does not use the technical document short name ([MS-xxxx]) to refer to the protocol.

  • Normative and informative citations cite only approved references.

  • Presents product behavior notes in the correct format <#> and links to the correct note in the Product Behavior appendix.

  • Uses the correct product name in references to specific products.

  • Does not contain pseudocode or code examples.

  • Footnotes are only used if you are not authoring in GenDox; any footnote text appears immediately after the table and clearly indicates which information it applies to.
