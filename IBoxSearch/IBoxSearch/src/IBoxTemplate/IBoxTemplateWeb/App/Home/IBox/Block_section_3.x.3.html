Standard text and formatting

Follow all authoring guidelines described in General Authoring Guidelines.

If there is no required initialization information for the protocol, do not delete this section. Instead, ensure that the first and only sentence in this section is "None."

Programming elements

Depending on the protocol or technology being specified, the Initialization section might contain descriptions of many different types of programming elements and related items. Follow the guidelines in the "Programming Elements Description Guidelines" section of the Windows Interoperability Standards Guide when describing programming elements and the Programming Element Formatting Guidelines when applying the correct formatting to programming elements.

Encryption keys and parameters

Descriptions of encryption keys should be formatted as:

A/An/The <key name> + <third-person, present-tense verb such as "encrypts/signs/verifies"> + <brief description that forms a complete sentence>. + <additional sentences as needed>.

For example:

The per-space encryption key encrypts all messages. It is derived from the per-space master key described in section 1.5. The per-space encryption key MUST be the result of a key derivation function.

Descriptions of encryption key parameters should be formatted as:

parameter: A/An/The + <noun phrase description>. + <additional sentences as needed>.

For example:

i_Key: The SHA-1 hash of the concatenation of the per-space master key and the Unicode string "MaskStringForTelespaceSecurityCipherKeys". This string MUST be hashed as a byte array. The zero terminator MUST NOT be included in the hash.

Required elements

  • If there is no required initialization information for the protocol, ensure that the first and only paragraph in this section is "None."

  • Explain when initialization occurs.

  • Indicate what actions are taken when the protocol is initialized and begins operation.

  • Specify protocol behaviors related to initialization.

Inappropriate elements

  • For block and sproc protocols, information about actions that are taken when the protocol "starts" in response to a higher-layer protocol

  • Initialization information that is not relevant to the current section 3.x

  • Code examples

  • Lowercase versions of normative terms; for example, must, should, and may

  • Diagrams, figures, or other art

  • Product names, unless those names are part of a protocol name

Checklist

  • If there is no required initialization information for the protocol, the first and only paragraph in this section is "None."

  • For block and sproc protocols, does not include actions that are taken when the protocol starts in response to a higher-layer protocol.

  • Explains when initialization occurs.

  • Describes programming elements according to the guidelines in the "Programming Elements Description Guidelines" section of the Windows Interoperability Standards Guide.

  • Applies the correct formatting to programming elements according to the Programming Element Formatting Guidelines

  • Lists all actions that are taken when the protocol is initialized and begins operation.

  • Organizes protocol behaviors and behavior requirements by trigger, not action.

  • Describes only those behaviors that are applicable to the role defined in the same section 3.x.

  • Any pseudocode conforms to the California Polytechnic State University standard for pseudocode.

  • Normative terms, as specified in RFC 2119, are used and capitalized correctly.

  • Does not use the lowercase versions of normative terms.

  • Does not use the technical document short name ([MS-xxxx]) to refer to the protocol.

  • Normative and informative citations cite only approved references.

  • Presents product behavior notes in the correct format <#> and links to the correct note in the Product Behavior appendix.

  • Does not contain code examples

  • Footnotes are only used if you are not authoring in GenDox; any footnote text appears immediately after the table and clearly indicates which information it applies to.
