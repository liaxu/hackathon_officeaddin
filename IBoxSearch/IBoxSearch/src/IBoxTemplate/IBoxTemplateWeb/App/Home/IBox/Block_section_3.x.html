Standard text and formatting

Follow all authoring guidelines described in General Authoring Guidelines.

Include this section only as applicable for a block protocol; if a particular role does not apply for the protocol, do not include a section for that role.

If there are no client or server roles to specify for a sproc protocol, the first and only paragraph in the corresponding section should be "None."

If the protocol can be described more easily by using a state machine diagram, provide such a diagram in this section. The diagram should use a subset of the UML 2 state machine diagram format, which is available in Visio 2003 and later. It is recommended that you omit from the diagram any actions and activities that occur within the protocol. Such actions and activities are typically complex and can best be specified in the text of subsections 3.1.x, where they additionally serve as a definitive reference for protocol behavior.

In the diagram, use a rectangle with rounded corners to represent a state.

To represent the initial pseudo-state, use a black solid circle.

To represent the final pseudo-state, use a black solid circle enclosed in another circle.

To represent transitions between states, use a directed arc. In the arc label, specify the trigger and, if applicable, a condition enclosed in square brackets.

Required elements

  • Include information about events that are surfaced from other protocols and cause the protocol to take some action, such as a connection failure in the underlying protocol which is signaled to this protocol as an error.

  • If the protocol is a sproc protocol, provide a Server Details section and a Client Details section.

  • If the protocol is a sproc protocol and it does not have a client or a server interface to specify, ensure that the first and only paragraph in the corresponding section is "None."

Inappropriate elements

  • Product names, unless those names are part of a protocol name

  • Code examples

  • References to platform-specific APIs

  • Information about state transitions

  • Information about events that are internal to the underlying transport protocol

  • Lowercase versions of normative terms; for example, must, should, and may

Checklist

  • If the protocol is a sproc protocol, contains a Server Details subsection and a Client Details subsection.

  • If the protocol is a sproc protocol and it does not have a client or a server interface to specify, the first and only paragraph in the corresponding section is "None."

  • Contains all required 3.x.x subsections.

  • Includes all relevant information about trigger events and protocol actions and activities.

  • Uses product behavior notes only where applicable to an implementation detail.

  • Normative terms, as specified in RFC 2119, are used and capitalized correctly.

  • Does not use the lowercase versions of normative terms.

  • Normative and informative citations cite only approved references.

  • Presents product behavior notes in the correct format <#> and links to the correct note in the Product Behavior appendix.

  • Includes accurate and illustrative diagrams, figures, or other art as necessary.

  • If a state diagram is provided, it conforms to the UML 2 state machine diagram format and uses the correct symbols.

  • Provides a figure number and caption for each diagram, figure, or graphic.

  • Any pseudocode conforms to the California Polytechnic State University standard for pseudocode.

  • Does not use the technical document short name ([MS-xxxx]) to refer to the protocol.

  • Does not contain code examples.

  • Does not contain references to platform-specific APIs.

  • Does not describe state transitions.

  • Does not provide details about underlying transport protocols.

  • Footnotes are only used if you are not authoring in GenDox; any footnote text appears immediately after the table and clearly indicates which information it applies to.

  • Does not contain product names, unless those names are part of a protocol name.
