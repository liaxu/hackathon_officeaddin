Standard text and formatting

Follow all authoring guidelines described in General Authoring Guidelines.

If there are no timer events for the protocol, do not delete this section. Instead, ensure that the first and only sentence in this section is "None." This is frequently the case.

If there are multiple timers, and therefore timer events, create a distinct subsection for each timer as shown in the example section below.

For each timer specified in section Timers, specify what happens when the timer expires. Specify the behavior in terms of:

  • Using the state in the Abstract Data Model (ADM) section

  • Sending of specific messages to specific destinations

  • Starting/resetting/stopping timers

  • Invoking events of this or other normative references

When sending messages, specify how any information is obtained that is required to form the messages. When invoking events of this or other normative references, specify how any information is obtained that is required to be passed.

When describing timer events, begin with "The <blah> event" followed by a third-person, present-tense verb such as "defines," followed by a brief description that forms a complete sentence ending in a period. For example:

The Call Response Timeout event completes the call with RESULT_TIMEOUT.

Required elements

  • If there are no timer events, ensure that the first and only paragraph in this section is "None."

  • Specify all of the actions that are triggered when each timer expires.

Inappropriate elements

  • Timer event information that is not relevant to the current section 3.x.

  • Lowercase versions of normative terms; for example, must, should, and may

  • Code examples

  • Pseudocode

  • Product names, unless those names are part of a protocol name

Checklist

  • If there are no timer events, the first and only paragraph in this section is "None."

  • If there are multiple timers, and therefore timer events, create a distinct subsection for each timer as shown in the Example section.

  • Discusses all timers that are specified in section Timers.

  • Indicates what happens when each timer expires.

  • Describes only those events that are applicable to the role defined in the same section 3.x.

  • Normative terms, as specified in RFC 2119, are used and capitalized correctly.

  • Does not use the lowercase versions of normative terms

  • Does not use the technical document short name ([MS-xxxx]) to refer to the protocol.

  • Normative and informative citations cite only approved references.

  • Presents product behavior notes in the correct format <#> and links to the correct note in the Product Behavior appendix.

  • Includes accurate and illustrative diagrams, figures, or other art as necessary.

  • Provides a figure number and caption for each diagram, figure, or graphic.

  • Does not contain pseudocode or code examples.

  • Footnotes are only used if you are not authoring in GenDox; any footnote text appears immediately after the table and clearly indicates which information it applies to.

  • Does not contain product names, unless those names are part of a protocol name.
