Standard text and formatting

Follow all authoring guidelines described in General Authoring Guidelines.

If there are neither versioning aspects of the protocol nor relevant information about capability negotiation, per the information below, do not delete this section. Instead, ensure that the first and only sentence in this section is "None."

Version information

The first part of this section should be a bulleted list of items with references and links to sections where versioning aspects of the protocol are specified in detail. Present list items in the following order:

  • Supported Transports - Does an aspect of the protocol go over SMB, raw IP, or something else?

  • Protocol Versions - Where does the protocol send version information?

  • Security and Authentication Methods - Which authentication protocols are supported? Where does the specification provide those details?

  • Localization - What happens if the protocol returns localized strings in a request or a response, such as an error string or user data? If the data is to be stored or displayed without context or negotiation of locale, the protocol has no localization-dependent behaviors. Otherwise, identify the localization-dependent behavior. An example is a response that specifies a locale identifier as the locale to which the message applies.

  • Capability Negotiation - Are there version dependencies? Are there cases where different products, product versions, or implementations within the same product demonstrate significantly different behavior?

If a specific item in the list does not apply to the protocol, omit it from the list but maintain the order above for any other items that do apply.

If the list contains more than one item, introduce it with the following sentence.

This specification covers versioning issues in the following areas:

If the list contains only one item, do not format it as a list. Instead, format it as a stand-alone paragraph with the appropriate information and a link to the relevant section in the specification.

Exchange: For Exchange Web Services SOAP protocols, if the protocol uses a date/time format, or if the MailboxCulture element is used to specify the culture of the mailbox, include this information in the Localization section.

Use the following text for the Security and Authentication Methods section:

This protocol relies on the web server that is hosting it to perform authentication.

In the Capability Negotiation section, provide information about how to get the target server version to determine which operations can be used, and which service endpoint to use.

Required elements

  • Summarize version and capability negotiation.

  • Provide an overview of negotiation issues, facilities, and handling.

  • Present all versioning aspects of the protocol in a list that conforms to the sequence identified above.

  • Link to all relevant sections and subsections.

  • If the list of versioning aspects contains more than one item, introduce it with "This specification covers versioning issues in the following areas:"

  • If the list of versioning aspects contains only one item, format the text as a stand-alone paragraph with the appropriate information; do not create a bulleted list with only one item.

  • For block and REST protocols, list any objects and packet exchanges that are involved in a negotiation.

  • For SOAP-based protocols, identify cases where a specific XML element or attribute is not available in every version of Microsoft products that implement the protocol.

  • For SOAP-based protocols, identify cases where the interpretation of specific XML schemas has changed.

  • For RPC-based protocols, adhere to the additional standards and guidelines as described above.

Inappropriate elements

  • Normative terms, as specified in RFC 2119; for example, MUST, SHOULD, and MAY

  • Lowercase versions of normative terms; for example, must, should, and may

  • Normative citations

  • Product behavior notes

  • Footnotes

  • Diagrams, figures, or other art

  • Tables

  • Code examples

  • Pseudocode

  • Product names, unless those names are part of a protocol name

Checklist

  • If versioning aspects and capability negotiation information does not apply to the protocol, the first and only paragraph in this section is "None."

  • References version and capability details in subsequent sections, instead of providing them in this section

  • Introduces the list of version information with the sentence "This document covers versioning issues in the following areas:"

  • Presents all versioning aspects of the protocol in a list that conforms to the sequence identified above

  • Lists contain at least two items; otherwise, the information is presented as a stand-alone paragraph, not a list

  • Does not use normative terms, as specified in RFC 2119

  • Does not use the lowercase versions of normative terms

  • For block protocols, lists any objects and packet exchanges

  • For SOAP-based protocols, identifies availability cases for specific XML elements or attributes that are not available in every version of Microsoft products that implement the protocol

  • For SOAP-based protocols, identifies changes to the interpretation of specific XML schemas

  • Does not use the technical document short name ([MS-xxxx]) to refer to the protocol

  • Uses informative citations sparingly

  • Does not contain normative citations

  • Does not contain pseudocode or code examples

  • Does not contain footnotes or product behavior notes

  • Does not contain diagrams, figures, or other art

  • Does not contain tables

  • Does not contain product names, unless those names are part of a protocol name
